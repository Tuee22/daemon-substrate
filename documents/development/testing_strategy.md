## TL;DR

- `daemon-substrate-test test unit` â€” pure logic, no cluster, runs anywhere.
- `daemon-substrate-test test lifecycle` â€” daemon-as-process, signals + readiness probes, no
  cluster.
- `daemon-substrate-test test integration` â€” executable 3x3 matrix; one invocation creates and
  tears down nine isolated `dst-test-*` clusters against real kind clusters.
- `daemon-substrate-test test lint` â€” governed-doc validation plus the direct
  `Daemon.Proto.*` import boundary.
- `daemon-substrate-test test all` â€” runs the above in order.
- `daemon-substrate-test` extends `hostbootstrap-core` via optparse-applicative
  (`runHostBootstrapCLI`). The operator entrypoint is `hostbootstrap cluster up`, driven by the
  thin Python bootstrapper that reads the **skeletal** `hostbootstrap.dhall`
  (`project`, `dockerfile`, `resources`). The rich project and per-case test Dhall are
  **generated by the binary**. See
  [../engineering/hostbootstrap_integration.md](../engineering/hostbootstrap_integration.md) and
  [../engineering/dhall_generation.md](../engineering/dhall_generation.md).
- **Coverage model** is the full executable **3Ã—3 matrix**: each of the three execution models
  (`Container`, `HostBinary`, `HostDaemon`) exercising each of three ML workflow archetypes â€”
  (a) continuous batched inference (â‰ˆ `infernix`), (b) finite SL / offline-RL training jobs
  (â‰ˆ `jitML`), and (c) continuous online RL (MinIO weight updates announced on Pulsar
  inference topics, with distinct training-vs-inference task messages routable to
  same-or-separate stateless engines). One integration invocation runs all nine cases.
- Each integration case runs under a `TestProfile`: a per-case data root `./.test_data/<case>/`
  and a test-scoped cluster name `dst-test-<model>-<archetype>`. The runner generates the
  per-case test Dhall, recursively invokes `hostbootstrap` to bring the case cluster up, deploys
  Harbor / Pulsar / MinIO plus a two-replica coordinator/orchestrator and exactly one worker,
  runs the case assertions, and tears the cluster down in a guaranteed `finally`. A `dst-test-`
  delete-guard ensures teardown can only target test clusters; production `.data` and any
  production cluster are never touched. See
  [../engineering/test_isolation.md](../engineering/test_isolation.md).
- The same compiled Haskell binary runs both long-running roles. Generated Dhall config selects
  coordinator/orchestrator vs worker behavior, including the Pulsar topics each role consumes
  and publishes. The coordinator/orchestrator may idempotently create missing lifecycle-policy
  topics; the worker / inference daemon must not create topics.

## What each command exercises

### `test unit`

The `daemon-substrate-unit` cabal stanza. Pure Haskell tests, no external services. Covers:

- protobuf encode / decode round-trips for every substrate-owned envelope (including the
  audit envelope)
- `WorkflowOwner` step-fold semantics against handcrafted event sequences
- `BootConfig` / `LiveConfig` / `LifecyclePolicy` Dhall decoders against fixture Dhall files
  in `test/unit/fixtures/`
- `Store` semantics over `Daemon.Test.FilesystemMinIO`
- `Daemon.MinIO.Cache` eviction policies under simulated key sets
- `Daemon.Consumer.consumerStep` dedup behavior over `Daemon.Test.FilesystemPulsar`
- reconciler tick correctness against filesystem-backed Pulsar + MinIO

### `test lifecycle`

The `daemon-substrate-lifecycle` cabal stanza. Daemon spawned as a real process (no cluster).
Covers:

- 7-phase lifecycle progression observable via `/readyz`
- SIGHUP â†’ `LiveConfig` reload visible in subsequent behavior
- SIGTERM / SIGINT â†’ graceful drain completes within `LiveConfig.drainDeadlineSeconds`

### `test integration`

The `daemon-substrate-integration` cabal stanza owns the full executable 3x3 matrix. It does
not rely on a preexisting cluster. For each of the nine cases it runs under a `TestProfile`
(data root `./.test_data/<case>/`, cluster name `dst-test-<model>-<archetype>`):

- generate the per-case test Dhall (see
  [../engineering/dhall_generation.md](../engineering/dhall_generation.md))
- recursively invoke `hostbootstrap` to bring up the case cluster for the selected execution
  model
- Harbor, Pulsar, and MinIO deployment
- harness image upload through Harbor without rebuilding the host/project artifact when an
  equivalent artifact is already available
- two-replica coordinator/orchestrator Deployment
- exactly one worker, either as an in-cluster Deployment or as a host-daemon process depending
  on the execution model
- workflow-specific assertions for the selected archetype
- cluster status verification
- **guaranteed `finally` teardown** of the case cluster before the next case, guarded by the
  `dst-test-` prefix so production is never a teardown target

The suite runs the same nine cases on Apple Silicon, Linux CPU, and Linux GPU hosts. The
physical host changes the build/run path and base-image selection, but it does not change which
matrix cells the integration suite executes. Production `.data` and any production cluster are
never touched; see [../engineering/test_isolation.md](../engineering/test_isolation.md).

When an integration case runs through the container execution model, the harness attaches the
current project container to Docker's `kind` network before using the internal kind kubeconfig.
For `HostDaemon`, the harness starts and owns the single foreground worker process for the
duration of that case.

The minimum assertion set for each case is:

- expected kind node count and node readiness for the selected execution model
- Harbor, Pulsar, and MinIO StatefulSet rollouts
- coordinator/orchestrator Deployment rollout with two Ready pods
- one worker ready in the expected placement
- Harbor image upload succeeded for the case cluster
- Harbor, Pulsar, and MinIO PVCs are bound for that cluster
- edge-port record fields for Pulsar, Pulsar admin, and MinIO
- workflow-specific Pulsar request -> coordinator/orchestrator -> worker -> result behavior
- clean `dst-test-` cluster teardown before the next matrix case

### `test lint`

The `daemon-substrate-haskell-style` cabal stanza. It enforces:

- governed-document metadata blocks, required broad-doc headings, relative Markdown link
  resolution, root README links to `documents/` and `DEVELOPMENT_PLAN/`, and phase-file
  `## Documentation Requirements` retention
- no direct `Daemon.Proto.*` imports outside the approved wire/boundary modules

### `test all`

Runs `lint`, then `unit`, then `lifecycle`, then `integration` in sequence. Stops at the
first failure. Because `integration` owns full cluster lifecycle, `test all` does not require a
preexisting kind cluster.

## Host and model obligations

Per [`../../DEVELOPMENT_PLAN/development_plan_standards.md` Â§ Q](../../DEVELOPMENT_PLAN/development_plan_standards.md),
the harness must validate two independent axes:

- **Physical hosts:** Apple Silicon, Linux CPU, and Linux GPU. The host changes the build/run
  path (in-container build + copy-out on Linux; native host-GHC build on Apple), the base-image
  selection, and whether the budget is enforced via a per-project Colima VM (Apple) or kind node
  cordoning (Linux). On a GPU host the CUDA `ensure` reconciler runs; the mock engine still does
  no CUDA computation.
- **Integration matrix:** `Container`, `HostBinary`, and `HostDaemon` each run the three
  workflow archetypes in one `daemon-substrate-test test integration` invocation, regardless
  of the physical host.

`Daemon.Test.Matrix` records the model/archetype cells. The integration runner uses that map
as the execution list, not just as a documentation audit.

The operator entrypoint is `hostbootstrap cluster up`, driven by the thin Python bootstrapper
that reads the skeletal `hostbootstrap.dhall`. The `daemon-substrate-test test ...` commands run
on the host binary the bootstrapper copies to `./.build/`. For HostDaemon cases, the harness owns
the single foreground worker process for the duration of that case. See
[../engineering/hostbootstrap_integration.md](../engineering/hostbootstrap_integration.md).

The mock engine performs no accelerator work. A GPU host validates the CUDA `ensure` reconciler,
the NVIDIA runtime prerequisite, and the CUDA-flavored base-image path, not CUDA computation.

## What this strategy does not cover

- Real ML model correctness â€” that is the consumer projects' obligation against their own
  matrices.
- WANâ†’MinIO weight hydration with real registries (HuggingFace, etc.) â€” the harness simulates
  this via `runFanInBootstrap` with a mock download function; real hydration is the consumer's
  deployment problem.
- Cross-substrate parity for consumer workloads â€” the substrate is parity-agnostic. The harness
  proves the same mock 3x3 matrix on each physical host; consumers prove real workload parity.
