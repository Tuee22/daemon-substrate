# Dhall Generation

**Status**: Authoritative source
**Supersedes**: N/A
**Referenced by**: [../README.md](../README.md), [hostbootstrap_integration.md](hostbootstrap_integration.md), [cluster_topology.md](cluster_topology.md), [test_isolation.md](test_isolation.md), [../reference/cli_surface.md](../reference/cli_surface.md), [../development/testing_strategy.md](../development/testing_strategy.md)

> **Purpose**: Define the binary-emitted Dhall schema/config contract — which Dhall tiers the
> `daemon-substrate-test` binary generates (`config schema` / `config render`), which tier is
> hostbootstrap-owned, and how generated project and per-case test Dhall feed the harness.

## TL;DR

- The `daemon-substrate-test` binary **emits its own Dhall schema** with
  `daemon-substrate-test config schema` and **renders config** with
  `daemon-substrate-test config render`. These are project verbs added on top of the
  `hostbootstrap-core` `config` command.
- **Only the skeletal tier is hostbootstrap-owned.** The repo-root `hostbootstrap.dhall`
  (`project`, `dockerfile`, `resources {cpu, memory, storage}`) is committed by hand and read by
  the Python bootstrapper.
- The **rich project Dhall** (worker/orchestrator roles + cluster bootstrap) and the
  **per-case test Dhall** are generated by the binary, not hand-written.
- Generation is deterministic and validated: the schema the binary emits is the schema its own
  decoders accept, so a rendered config that round-trips through `config render` is guaranteed to
  decode at `service` / `cluster` time.

## Three tiers, two producers

| Tier | Contents | Producer | Consumer |
|------|----------|----------|----------|
| **Skeletal** | `project`, `dockerfile`, `resources {cpu, memory, storage}` | committed by hand | Python bootstrapper |
| **Project** | worker + orchestrator role records, cluster bootstrap topology | `daemon-substrate-test config render` | the binary's `service` / `cluster` verbs |
| **Per-case test** | one record per matrix case (model × archetype), test-scoped names, `.test_data/<case>` paths | `daemon-substrate-test` integration runner | recursive `hostbootstrap` invocation per case |

See [hostbootstrap_integration.md](hostbootstrap_integration.md) for how the tiers map onto the
Python / `hostbootstrap-core` / binary ownership boundary.

## `config schema`

`daemon-substrate-test config schema` prints the Dhall type the binary's decoders accept for the
project and per-case tiers. The schema is owned by the binary because the binary owns the
decoders: there is no separate hand-maintained schema file that can drift from the Haskell types.
Consumers and CI can capture the emitted schema to diff against a committed snapshot when a phase
calls for schema-stability checks.

## `config render`

`daemon-substrate-test config render` materializes a concrete project-tier Dhall config from the
binary's defaults plus the skeletal `hostbootstrap.dhall` inputs (notably `project` and
`resources`). The rendered config is what `service --role worker|orchestrator` and the
`cluster ...` reconcilers decode. Rendering is idempotent: re-rendering with the same inputs
produces byte-identical output, so a generated config can be regenerated rather than edited.

## Per-case test Dhall

The integration runner generates one test-tier Dhall per matrix case before invoking
`hostbootstrap` for that case. Each generated record carries:

- the execution model and workflow archetype for the case
- the **test-scoped cluster name** `dst-test-<model>-<archetype>`
- the **per-case data root** `./.test_data/<case>/`
- the role topology and edge-port policy for that case

These records are never committed; they are emitted into the per-case `.test_data/<case>/`
workspace and consumed by the recursive `hostbootstrap` invocation. The `ClusterProfile`
distinction (production vs test) that drives name and path selection lives in
[test_isolation.md](test_isolation.md).

## Why generation rather than committed Dhall

- **No drift.** The schema and the rendered config both come from the binary's own decoders, so a
  rendered config cannot reference a field the decoder does not understand.
- **Nine cases without nine files.** The 3x3 matrix needs a distinct config per case; generating
  them keeps the repo from carrying nine near-identical hand-edited Dhall files.
- **Production safety.** Production-tier paths/names and test-tier paths/names come from the same
  generator keyed on `ClusterProfile`, so the test path can never accidentally resolve to a
  production name. See [test_isolation.md](test_isolation.md).
