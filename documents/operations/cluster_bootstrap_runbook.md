> **Purpose**: Operator-facing reference for cluster lifecycle — the thin Python bootstrapper
> (`hostbootstrap cluster ...`), the per-project resource budget and Colima/kind cordoning, the
> inner `daemon-substrate-test cluster ...` reconcilers, and the lifecycle phases to expect.

## TL;DR

- `hostbootstrap cluster up` is the operator entry. The **thin Python bootstrapper** reads the
  **skeletal** `hostbootstrap.dhall` (`project`, `dockerfile`, `resources`), reaches a fail-fast
  host minimum, ensures Docker (a per-project **Colima** VM on Apple), builds the project
  container with the `check-code` gate, copies `daemon-substrate-test` to `./.build/`, and execs
  the inner `cluster up`.
- The per-project **resource budget** (`resources {cpu, memory, storage}`) is enforced via a
  per-project Colima VM on Apple and **kind node resource cordoning** on Linux.
- `./.build/` always holds a host-runnable `daemon-substrate-test` binary.
- The rich project Dhall (roles + cluster bootstrap) is **generated by the binary**, not
  hand-written. See [../engineering/dhall_generation.md](../engineering/dhall_generation.md).
- `hostbootstrap cluster down` / `hostbootstrap cluster delete` reconcile cluster absence and
  preserve `./.data/`.
- `hostbootstrap` installs no launchd/systemd units and creates no reboot-persistent containers.
  Run `hostbootstrap cluster up` after each reboot.
- The nine-case `test integration` matrix runs on isolated `dst-test-*` clusters and never
  touches a production cluster or `./.data/`. See
  [../engineering/test_isolation.md](../engineering/test_isolation.md).

## Ownership Boundary

- **Python bootstrapper (thin)**: read the skeletal `hostbootstrap.dhall`; fail-fast host
  minimums; `ensure docker` (per-project Colima VM on Apple); build the project container; copy
  the binary to `./.build/`; exec the requested verb.
- **`hostbootstrap-core` (library)**: host-tool resolution, the `ensure` reconcilers, substrate
  detection, cluster-lifecycle semantics + kind resource cordoning, and the optparse command
  tree.
- **`daemon-substrate-test` (binary)**: generate project/test Dhall and emit its own schema; kind
  create, Helm install of Harbor / Pulsar / MinIO, ConfigMap render, Deployment apply, MinIO
  bucket seeding, edge-port discovery, lifecycle phase transitions, and readiness checks.

See [../engineering/hostbootstrap_integration.md](../engineering/hostbootstrap_integration.md)
for the full boundary statement.

## Host preparation

```bash
hostbootstrap ensure docker
```

`ensure docker` reconciles the Docker runtime: on Apple it provisions a per-project Colima VM
sized to the `resources` budget; on Linux it verifies native Docker reachability. Other tools are
reconciled the same way (`ensure ghc`, `ensure cuda`, `ensure tart`), each failing fast on the
wrong host.

## Bring-up

```bash
hostbootstrap ensure docker
hostbootstrap cluster up
```

`cluster up` runs the Python flow (fail-fast minimum → ensure Docker → build → copy binary to
`./.build/` → exec) and the inner `daemon-substrate-test cluster up` reconciles, in order:

1. **Kind cluster**: create if missing under the active `ClusterProfile` name; treat an existing
   cluster as a successful no-change action; verify the node topology and apply the
   budget-derived **resource cordoning** to kind nodes (Linux).
2. **Manual storage**: install the `daemon-substrate-manual` StorageClass and provision durable
   PVs backed by the profile's kind data mount.
3. **Image publication**: reuse or build the harness artifact as needed, deploy Harbor, and
   upload the harness image into the cluster's Harbor registry.
4. **Helm dependencies**: build or refresh Harbor, Pulsar, and MinIO chart dependencies.
5. **Dependency readiness**: wait for Harbor, Pulsar, and MinIO StatefulSets to become ready.
6. **Pulsar bootstrap**: create the harness tenant, namespace, and topics idempotently.
7. **MinIO bootstrap**: create the harness buckets and seed mock blobs.
8. **ConfigMaps**: render orchestrator and worker ConfigMaps from the generated project Dhall.
9. **Orchestrator Deployment**: roll out and wait for readiness.
10. **Worker**: roll out one in-cluster worker for `Container` / `HostBinary`, or run one
    harness-owned host-native worker process for `HostDaemon`.
11. **Edge port discovery / forwarding**: pick and persist Pulsar, Pulsar admin, and MinIO edge
    ports; host-native paths use those records to reach in-cluster services.

## Ready Definition

The cluster is `Ready` when:

1. Kind node count matches the selected model topology and stays within the budget cordon.
2. Pulsar admin is reachable on the chosen edge port.
3. Every MinIO bucket named in the generated `LifecyclePolicy` exists.
4. The orchestrator Deployment is `2/2` Ready.
5. The worker is Ready: one in-cluster worker for `Container` / `HostBinary`, one host-native
   process for `HostDaemon`.
6. `runReconciler` has completed at least one full tick.

`daemon-substrate-test test integration` uses this readiness contract inside each of its nine
`dst-test-*` cases, generating per-case Dhall and tearing each cluster down in a guaranteed
`finally`. See [../reference/cli_surface.md](../reference/cli_surface.md),
[../development/testing_strategy.md](../development/testing_strategy.md), and
[../engineering/test_isolation.md](../engineering/test_isolation.md).

## Status

```bash
./.build/daemon-substrate-test cluster status
```

`cluster status` is read-only and does not mutate Kubernetes state, repo-local state, or the
chosen edge port.

## Teardown

```bash
hostbootstrap cluster down
hostbootstrap cluster delete
```

`cluster down` reconciles cluster absence while preserving repo-local durable state. `cluster
delete` is the thorough inner teardown path and still preserves `./.data/`. The test path can
only delete `dst-test-`-prefixed clusters and only reconciles `./.test_data/<case>/` workspaces.

Preserved state:

- `./.data/` — durable production cluster state and PV-backing files
- `./.build/` — the host-runnable binary
- installed host prerequisites and local Docker layer cache

## Reboot Policy

No `hostbootstrap` lifecycle artifact survives reboot as an automatically restarted service.
Docker invocations are one-shot and host worker processes are caller-owned. After reboot:

```bash
hostbootstrap cluster up
```

Operators who want boot-time automation can create their own launchd/systemd unit outside
`hostbootstrap`.

## Kubeconfig

The repo-local kubeconfig is the only authoritative handle to the harness cluster:

| Profile | kubeconfig |
|---------|------------|
| `ProductionProfile` | `./.data/runtime/daemon-substrate.kubeconfig` (container) or `./.build/daemon-substrate.kubeconfig` (host-native) |
| `TestProfile` | `./.test_data/<case>/daemon-substrate.kubeconfig` |

Neither path mutates the operator's global kubeconfig. To use `kubectl` directly:

```bash
kubectl --kubeconfig ./.build/daemon-substrate.kubeconfig get pods -A
```
